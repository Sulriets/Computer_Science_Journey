# Capítulo 01: ¿Por qué GitHub usa Markdown?

Si llegaste hasta aquí felicidades 🎊 . Estamos un paso más cerca en este largo camino. Pero sin presiones, Roma no se contruyó en un día. 😜 

Comencemos a responder esta pregunta por partes. Porque, tal vez la pregunta más sencilla y a la vez más importante sería comenzar con:

## ¿Qué es Github? 🤔 

Lo sé, parece obvio. Pero al mismo tiempo, no lo es. En mí experiencia, mi primera vez que abrí esta plataforma, casí hasta dió miedo con la cantidad de cosas que no entendía 😨. Sin embargo, aquello es parte de la experiencia (Y no.. no es traumática 🤣). 

Piensalo por un momento.. ¿Qué es Github?
<details><summary>Respuesta</summary>GitHub es un servicio basado en la Nube (Esta online). Aloja un repositorio llamado Git (VCS). En donde se realizan cambios y seguimiento detallado. Imagina un trabajo en grupo online en donde todos los participantes (llamados colaboradores) comparten y realizan cambios a proyectos. 😎</details>

Sabiendo esto, antes de GitHub, ya existía:

* HTML
* Word
* PDF
* Texto plano

Y aquí viene la pregunta del millón de dólares.. Sí existian todos esos diferentes tipos para escribir texto. ¿Por qué GitHub terminó usando Markdown? 💸

---

Para responder a esta pregunta pasaremos por `Siete conceptos`:

### Primer concepto:

Este ya lo respondimos anteriormente. Los programadores, escriben código, no libros (pero aclaro, los buenos programadores sí escriben mucha documentación). Es cómo pedirle a un maestro pintor 🫟, que en vez de pintar una casa, te haga un cuadro en cubismo y con efectos de luz 👨🏻‍🎨🎨.

Sí te pusieras en su lugar. Enfocado en hacer tu trabajo. ¿Abandonarías el editor de código para documentar en Word lo que estas haciendo, cada dos líneas?

### Segundo concepto:

Siguiendo con la idea. Todo proyecto, no importa si es pequeño o grande, tiene que explicar y responder preguntas. 

| PREGUNTAS CLAVE | RESPUESTA |
| :--- | ---: | 
| ¿Qué hace? | Probablemente la pregunta más importante | 
| ¿Cómo se instala? | Aspecto técnico, sí no lo especificas, simplemente no se instala | 
| ¿Cómo se usa? | Aspecto práctico, sí nadie sabe usarlo. No recibes información de regreso | 
| ¿Quién lo hizo? | Tal vez suene egocéntrico de primeras, pero el credito se puede quitar | 

Aun peor sí solo subes archivos así: `main.py` o `app.py` o `utils.py` ¿Qué crees que pasé? 🤯

### Tercer concepto:

¿Por qué no usamos HTML?

Respuesta corta. Somos prácticos 🤣. Solo recuerda como era `HTML vs Markdown` en el capítulo anterior. Es terriblemente lento escribir en HTML. 

### Cuarto concepto:

Tranquilo este aspecto si es técnico a diferencia del anterior. Markdown es reconocido por ser **legible** incluso sin renderizarse. 

Coloquemonos en el peor futuro posible, ☢️ casi post-apocaliptico.. GitHub desaparece 😱. 

Pero milagro celestial 🪽, puedes abrir un README en un Block de notas. Salvación. 

Esto es parte de la filosofía de Gruber. 

### Quinto concepto:

Esta conectado con el anterior. GitHub renderiza Markdown automáticamente. No necesitas un convertidor. Subes un README y este maravilloso programa hará el resto. Fácil y bonito. 🦋

### Sexto concepto:

Git entiende texto. Dejame insistir nuevamente.. TEXTO. No estoy hablando de Word (esto no es solamente texto) o pdf (tampoco, no es texto) o imágenes (🙄). TEXTO. Y ¿Qué es Markdown?

Respuesta corta: es Texto plano ➡️ Git trabaja con texto ➡️ gracias a eso compará línea por línea ➡️ por eso un README en Markdown funciona.

### Séptimo concepto:

Markdown desde que nació en 2004 hasta el día de hoy, ya es un estándar, no solo para GitHub. También para:
* GitLab
* Azure DevOps
* Obsidian
* Jupyter
* MkDocs
* etc.

## Pregunta para refleccionar: 🤔 Si GitHub hubiese elegido Word para documentar... ¿Cómo sería?

Entonces querido Lector. Después de todo este recorrido, la respuesta ya no parece misteriosa. GitHub eligió Markdown justamente porque permite escribir codigo y documentar sin interrumpir el flujo de trabajo.
